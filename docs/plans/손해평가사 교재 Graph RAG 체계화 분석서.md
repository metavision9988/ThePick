## 3. Graph RAG 4-Level 메타데이터 스키마

### 3.1 계층 구조 정의

```
LV1: 보장방식 (Insurance Type)
 └── LV2: 품목/작물군 (Crop Category)
      └── LV3: 조사 종류 (Investigation Type)
```

## 4. Graph 노드-엣지 설계

### 4.1 노드(Node) 유형

교재에서 추출 가능한 노드는 **6가지 타입**으로 분류된다:

| 노드 타입       | 설명      | 예시                           | 추정 수량 |
| --------------- | --------- | ------------------------------ | --------- |
| `CONCEPT`       | 개념/정의 | 순수위험, 대수의법칙, 실손보상 | ~150개    |
| `INSURANCE`     | 보장방식  | 적과전 종합위험, 수확감소보장  | ~8개      |
| `CROP`          | 품목      | 사과, 벼, 인삼, 느타리버섯     | ~70개     |
| `INVESTIGATION` | 조사 종류 | 착과수조사, 경작불능조사       | ~20개     |
| `FORMULA`       | 산식      | 피해율, 보험금, 감수과실수     | ~80개     |
| `LAW`           | 법조문    | 농어업재해보험법 제11조        | ~50개     |
| `TERM`          | 용어      | 자기부담비율, 미보상비율       | ~120개    |

**추정 총 노드 수: ~500개**

### 4.2 엣지(Edge) 유형

| 엣지 타입                | 의미             | 예시                                    |
| ------------------------ | ---------------- | --------------------------------------- |
| `APPLIES_TO`             | 보장방식 → 품목  | INS-01 → 사과                           |
| `REQUIRES_INVESTIGATION` | 보장방식 → 조사  | INS-01 → 착과수조사                     |
| `PREREQUISITE`           | 조사 간 선후관계 | 착과수조사 → 낙과피해조사               |
| `USES_FORMULA`           | 조사 → 산식      | 착과피해조사 → 감수과실수 산출          |
| `DEPENDS_ON`             | 산식 → 변수      | 보험금 = f(감수량, 자기부담비율)        |
| `GOVERNED_BY`            | 절차 → 법조문    | 손해평가반 구성 → 손해평가요령 제8조    |
| `DEFINED_AS`             | 용어 → 정의      | 실제결과주수 → "실제 결과 맺는 나무 수" |
| `EXCEPTION`              | 일반규칙 → 예외  | 피해구성조사 → "경미시 생략 가능"       |
| `TIME_CONSTRAINT`        | 시간 조건        | 낙엽률조사 → "6월1일~10월"              |

**추정 총 엣지 수: ~1,500개**

### 4.3 핵심 Graph 패턴 (교재에서 반복되는 구조)

교재 전체를 관통하는 **5단계 손해평가 흐름**이 모든 보장방식에서 반복된다:

```
[사고접수] → [피해사실확인] → [조사방법 선택] → [현지조사 실행] → [보험금 산정]
     ↑              ↑                ↑                  ↑               ↑
   TRIGGER      INVESTIGATION    ROUTING LOGIC      FORMULA          CALCULATION
   (재해종류)    (보장재해확인)    (조건부 트리)     (산식 적용)       (룰 엔진)
```

**이 흐름이 보장방식마다, 품목마다, 재해종류마다 변수가 달라진다.**
이것이 바로 단순 벡터 검색이 실패하고 Graph RAG가 필수인 이유다.

### 7.4 방어적 설계 보완 (v2 — ARCHITECT × BREAKER 합동 리뷰 반영)

외부 리뷰에서 제기된 3가지 사각지대를 검토한 결과, **모두 타당**하다.
아래에 각 지적 사항의 타당성 근거와 반영 내역을 기록한다.

#### 7.4.1 시간축의 누락 (Temporal Blind Spot) — ✅ 타당, 반영 완료

**타당성 근거:**
교재 표지에 "2025년 12월 31일 기준 관련법령"이라고 명시되어 있다.
손해평가요령은 매년 농금원에서 개정 고시하며, 인정피해율 계수(F-06의 `1.0115`, F-07의 `0.9662`)조차 연구용역 결과에 따라 변경될 수 있다.
노드를 덮어쓰면 과거 기출문제의 맥락이 소실되어 삼각 교차 검증이 불가능해진다.

**반영 내역:**

- `knowledge_nodes`에 `version_year`, `superseded_by` 필드 추가 (7.2 스키마 반영 완료)
- `formulas`에 `version_year`, `superseded_by` 필드 추가
- `knowledge_edges`에 `is_active` 필드 추가 + `SUPERSEDES` 엣지 타입 추가
- **Hard Rule: UPDATE 금지.** 개정 시 반드시 새 노드 생성 → `SUPERSEDES` 엣지로 연결 → 구 엣지 `is_active=0`

```
[2026 노드: F-06 단감 인정피해율 = 1.0115×낙엽률 – 0.0014×경과일수]
    ↑ SUPERSEDES
[2025 노드: F-06 단감 인정피해율 = (구 계수 적용)]  ← is_active=0
```

#### 7.4.2 진실의 충돌과 우선순위 (Source of Truth Conflict) — ✅ 타당, 반영 완료

**타당성 근거:**
교재 p.400에서 "손해평가요령에 따라 공정하고 객관적으로 손해평가를 하여야 하며"라고
법률 원문을 인용하고 있다. 교재 설명은 법령의 해설이지 법령 자체가 아니다.
실제로 교재 텍스트와 부록 법령 원문 사이에 표현 차이가 존재하며,
시험에서는 법령 원문이 우선한다.

**반영 내역:**

- `knowledge_nodes`에 `truth_weight` 필드 추가 (7.2 스키마 반영 완료)
- 가중치 체계:

| truth_weight | 노드 타입                | 근거                            |
| ------------ | ------------------------ | ------------------------------- |
| 10           | LAW (법조문 원문)        | 법률 > 모든 것                  |
| 8            | FORMULA (산식)           | 손해평가요령 별표에서 직접 도출 |
| 7            | INVESTIGATION (조사방법) | 손해평가요령 본문에서 도출      |
| 5            | CONCEPT (교재 설명)      | 교재 저자의 해설                |
| 3            | TERM (용어 정의)         | 참조용                          |

- **AI 응답 조합 규칙:** RAG 검색 결과를 LLM에 주입할 때 `truth_weight` 내림차순 정렬.
  LAW 노드와 CONCEPT 노드가 충돌하면 LAW를 채택하고 CONCEPT는 "참고" 레이블로 표시.

#### 7.4.3 인간 검증(G5.5)의 병목 현상 — ✅ 타당, Phase 0 태스크에 추가

**타당성 근거:**
500개 노드 + 1,500개 엣지를 JSON/YAML 파일로 검수하는 것은 인간에게 인지적 과부하.
Admin CMS(Phase 1 Epic 1.3)를 기다려서는 데이터 정제가 시작조차 불가능.
**검증 도구 없는 Graph RAG는 구축 자체가 불가능하다.**

**반영 내역:**

- Phase 0 태스크에 **"Graph Visualizer & Approver (경량 검증 도구)"** 추가
- 기술 스택: Astro + D3.js (또는 React Force Graph) — 단일 HTML 파일로도 가능
- 최소 기능:
  1. 노드/엣지를 시각적으로 탐색 (Force-directed graph)
  2. 노드 클릭 → 상세정보 패널 (description, formula, page_ref)
  3. status를 `draft → review → approved` 로 변경하는 버튼
  4. 필터: LV1/LV2/LV3별 하위 그래프만 표시
- **이 도구는 수험생 프론트엔드보다 먼저 구축한다.**
