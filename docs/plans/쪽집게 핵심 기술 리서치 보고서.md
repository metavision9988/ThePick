## 1. Cloudflare D1 (데이터베이스)

### 판정: ⚠️ 채택 (주의사항 있음)

**우리에게 맞는 이유:**

우리 워크로드는 90%+ 읽기입니다. 수험생이 기출 풀이, 복습 카드, 해설 조회를 하는 것이 주 사용이고, 쓰기는 학습 진도 저장과 관리자 콘텐츠 승인 정도입니다. D1은 read-heavy 워크로드에서 sub-10ms 읽기 성능을 보여주며, Workers와 네이티브 바인딩으로 연결되어 연결 풀링이나 콜드 스타트 문제가 없습니다.

**검증된 수치:**

- DB당 10GB 한도 — 우리 전체 데이터(~1,140 노드 + ~2,600 엣지 + ~150 상수 + ~825 기출)는 추정 100MB 미만. 한도의 1%도 안 됨
- SELECT(인덱스 있음): < 1ms SQL duration
- INSERT/UPDATE: 수 ms (내구성 보장을 위해 여러 위치에 영속화)
- 단일 스레드: 쿼리를 한 번에 하나씩 처리. 동시 쓰기가 많으면 병목

**잠재 위험 + 완화 전략:**

| 위험                     | 심각도                     | 완화                                                            |
| ------------------------ | -------------------------- | --------------------------------------------------------------- |
| 10GB/DB 제한             | 낮음 (우리 데이터 < 100MB) | 다른 시험 확장 시 per-exam DB 샤딩                              |
| 단일 스레드 쓰기         | 중간                       | 학습 진도는 배치 동기화(5분 간격), 실시간 쓰기 최소화           |
| Vendor lock-in           | 중간                       | Drizzle ORM이 DB 교체를 추상화. 탈출구: Turso(libSQL 호환)      |
| D1 외부 접근 불가        | 낮음                       | 관리 도구는 Workers API 경유. 직접 SQL 클라이언트 접속 불가     |
| 마이그레이션 도구 미성숙 | 중간                       | Drizzle Kit으로 마이그레이션 관리. wrangler d1 migrations apply |

**기각된 대안:**

- Turso(libSQL): 임베디드 레플리카로 마이크로초 수준 읽기 가능하나, 우리는 이미 Cloudflare 생태계(Workers+KV+R2)에 올인. Turso 추가 시 인프라 이중화
- Supabase(Postgres): 서버리스에서 연결 풀링 문제, Workers 호환 추가 설정 필요, 비용 $25+/월
- PlanetScale: 2024년 무료 티어 폐지 이후 비용 부담, MySQL 기반이라 SQLite 친화적 스택과 불일치

**결론:** D1은 우리 워크로드(read-heavy, < 1GB, Cloudflare 생태계)에 최적. 다른 시험 확장 시 per-exam 샤딩 전략만 미리 설계해두면 됨.

## 2. Cloudflare Vectorize (벡터 검색)

### 판정: ⚠️ 채택 (주의사항 있음)

**우리에게 맞는 이유:**

Graph RAG의 맥락 계층(해설/부연설명/사례)을 벡터 유사도로 검색하는 용도. Workers 바인딩으로 네트워크 홉 없이 접근 가능. D1 메타데이터 필터 → Vectorize 유사도 검색 → truth_weight 정렬의 3단계 파이프라인에서 2단계 담당.

**검증된 수치:**

- 인덱스당 5백만 벡터 한도 — 우리는 ~1,140 벡터면 충분. 한도의 0.02%
- Worker 바인딩 쿼리: 엣지에서 처리되어 저지연
- HNSW 인덱싱 + Product Quantization으로 근사 최근접 이웃 검색
- 메타데이터 필터링: 문자열 인덱스 최초 64바이트까지

**잠재 위험 + 완화 전략:**

| 위험                          | 심각도 | 완화                                                                   |
| ----------------------------- | ------ | ---------------------------------------------------------------------- |
| 메타데이터 필터 64바이트 제한 | 중간   | LV1/LV2/LV3 ID는 짧은 코드(INS-01, CROP-APPLE). 64바이트 충분          |
| 근사 검색 정확도              | 중간   | Graceful Degradation(유사도 < 0.60 → 거부)로 잘못된 결과 차단          |
| dimension 변경 불가           | 높음   | 인덱스 생성 시 차원 고정. **임베딩 모델 선택을 Phase 0에서 확정 필수** |
| 비용 예측 어려움              | 낮음   | Workers Paid $5/mo에 Vectorize 포함. 우리 규모에서 추가 비용 미미      |

**핵심 결정 필요 (Phase 0에서):**

임베딩 모델 선택이 Vectorize 인덱스 차원을 결정하며, 한번 정하면 변경 불가. 선택지:

- Workers AI `bge-small-en-v1.5` (384차원) — 무료, Cloudflare 네이티브, 영어 최적화
- Workers AI `bge-m3` (1024차원) — 다국어 지원, 한국어 성능 우수할 가능성
- OpenAI `text-embedding-3-small` (1536차원) — 범용, 비용 발생

**→ 한국어 교재 기반이므로 다국어 모델 성능 벤치마크가 Phase 0 PoC에서 필수.**

## 3. Astro + React Islands (프론트엔드)

### 판정: ✅ 채택 확정

**우리에게 맞는 이유:**

우리 PWA는 "학습 콘텐츠 소비 80% + 인터랙션 20%" 구조입니다. 기출 해설 조회, 플래시카드 표시, 대시보드 차트 — 대부분 정적 렌더링이 가능하고, 스와이프 제스처나 타이머 같은 인터랙티브 부분만 React Island로 처리하면 됩니다.

**검증된 수치:**

- Astro는 Next.js 대비 JS 번들 50~70% 절감 (zero-JS 기본)
- 콘텐츠 중심 사이트에서 Lighthouse 성능 점수 95+ 달성 용이
- 빌드 속도 Next.js 대비 약 3배 빠름 (1000페이지 기준 18초 vs 52초)
- React Islands에서 `client:visible`로 뷰포트 진입 시에만 hydration — 초기 로드 최소화

**Astro의 약점 + 완화:**

| 약점                           | 심각도 | 완화                                                                    |
| ------------------------------ | ------ | ----------------------------------------------------------------------- |
| 복잡한 인터랙션 제한           | 중간   | 학습 앱 수준에서는 Islands로 충분. 대시보드, 스와이프, 타이머 모두 가능 |
| 인증/세션 관리 약함            | 중간   | 인증은 Workers API에서 JWT 처리. Astro는 토큰 전달만                    |
| Islands 과다 사용 시 성능 저하 | 낮음   | Island 수를 페이지당 3~5개 이하로 제한하는 내부 규칙                    |
| Next.js 대비 생태계 작음       | 낮음   | React 컴포넌트 그대로 사용 가능. shadcn/ui, recharts 등 호환            |

**기각된 대안:**

- Next.js: SSR 서버 필요, Vercel 종속, 번들 크기 큼. 우리는 Cloudflare Pages 배포
- SvelteKit: 생태계 작음, React 개발자 풀 활용 불가, shadcn/ui 미지원
- Remix: 서버 중심 철학이 오프라인 PWA와 충돌

## 4. Drizzle ORM

### 판정: ✅ 채택 확정

**우리에게 맞는 이유:**

Cloudflare Workers + D1 조합에서 Drizzle은 사실상 유일한 실용적 선택입니다.

**핵심 장점:**

- D1 네이티브 지원 (별도 어댑터/프록시 불필요)
- 번들 ~7.4KB (Prisma ~40KB+ + 바이너리). Workers 10MB 번들 제한에서 중요
- TypeScript 스키마 = 런타임 타입 = 마이그레이션 소스. 코드 생성 단계 없음
- SQL에 가까운 쿼리 빌더 — 생성되는 SQL을 예측 가능

**Prisma 기각 사유:**

- Workers에서 네이티브 실행 불가 → Prisma Accelerate(유료 프록시) 필요
- v7에서 Rust 엔진 제거했으나 여전히 번들 크기 3~5배 차이
- D1 지원은 아직 공식적으로 제한적
- 코드 생성(`prisma generate`) 스텝이 개발 흐름에 마찰 추가

**Drizzle 주의사항:**

| 주의                                       | 완화                                                                  |
| ------------------------------------------ | --------------------------------------------------------------------- |
| 마이그레이션 도구가 Prisma보다 단순        | `drizzle-kit generate` + `push`로 충분. 복잡한 마이그레이션은 raw SQL |
| 릴레이션 쿼리가 Prisma보다 직관적이지 않음 | 우리 쿼리 대부분 단순 SELECT+JOIN. 복잡한 중첩 릴레이션 없음          |
| 1.0 정식 출시 미완 (안정화 진행 중)        | 2025년 이후 API 안정, 주간 다운로드 500만+. 프로덕션 사용 충분        |

## 5. math.js AST 파서 (Formula Engine)

### 판정: ⚠️ 채택 (보안 강화 필수)

**우리에게 맞는 이유:**

125개 보험금 산식을 안전하게 실행해야 합니다. math.js는 `parse()` → AST → `evaluate()`로 동적 코드 실행(`eval`, `new Function`) 없이 수식을 연산합니다.

**보안 강점:**

- math.js v4+는 내부적으로 JavaScript `eval()` 사용하지 않음
- AST를 통해 허용된 연산만 실행 가능
- `import`, `createUnit`, `evaluate` 등 위험 함수를 비활성화할 수 있음

**반드시 적용해야 할 보안 강화:**

```typescript
// 위험 함수 비활성화 (Hard Rule #9 구현)
const math = create(all);
math.import(
  {
    import: () => {
      throw new Error('disabled');
    },
    createUnit: () => {
      throw new Error('disabled');
    },
    evaluate: () => {
      throw new Error('disabled');
    },
    parse: () => {
      throw new Error('disabled');
    },
    simplify: () => {
      throw new Error('disabled');
    },
    derivative: () => {
      throw new Error('disabled');
    },
    resolve: () => {
      throw new Error('disabled');
    },
  },
  { override: true },
);
```

**잠재 위험 + 완화:**

| 위험                   | 심각도   | 완화                                                                         |
| ---------------------- | -------- | ---------------------------------------------------------------------------- |
| 번들 크기 (~170KB min) | 중간     | Workers 10MB 한도 내 가능. 트리쉐이킹으로 사용 함수만 포함                   |
| 부동소수점 정밀도      | **높음** | `1.0115 × 0.45`같은 계수 곱셈에서 오차 발생 가능. 교재 예시값 100% 대조 필수 |
| 과도한 재귀/무한루프   | 중간     | 산식 복잡도 제한 (AST 깊이 10 이하), Workers CPU 타임아웃(30초)이 방어선     |

**검토한 대안:**

- fparse: 경량(~15KB), AST 지원, 빠름. 단 math.js만큼 풍부한 수학 함수 없음. 우리 산식이 단순 사칙연산+거듭제곱 수준이면 검토 가치 있음
- 자체 파서: 통제력 최고, 개발 비용 과다. 기각
- hot-formula-parser: Excel 호환 목적. 우리 용도와 불일치

**결론:** math.js 채택하되, 위험 함수 비활성화 + 부동소수점 정밀도 검증을 Phase 0 Formula Engine PoC에서 반드시 확인. 번들 크기가 Workers 한도를 압박하면 fparse로 교체 검토.

## 6. FSRS-5 (간격반복 알고리즘)

### 판정: ✅ 채택 확정

**SM-2 대비 FSRS-5의 우위:**

- SM-2는 1987년 설계, 개인화 없음. 모든 사용자에게 동일한 간격
- FSRS-5는 머신러닝 기반으로 개인별 망각 곡선을 모델링
- Anki 커뮤니티에서 대규모 비교 연구: FSRS가 SM-2 대비 기억 효율 30~50% 개선

**잠재 위험 + 완화:**

| 위험                                      | 심각도   | 완화                                                          |
| ----------------------------------------- | -------- | ------------------------------------------------------------- |
| Python 참조 구현 → TypeScript 포팅 정확도 | **높음** | M17-T01: Python 참조와 100% 동일 결과 검증 (100% 필수 테스트) |
| 서버+클라이언트 이중 실행 동기화          | 중간     | 동기화 시 서버 결과 우선. 오프라인 로컬 계산은 "임시 스케줄"  |
| FSRS 파라미터 튜닝                        | 낮음     | 기본 파라미터로 시작, 사용자 데이터 축적 후 조정              |
| 초기 데이터 부족 (콜드스타트)             | 중간     | confusion_level=danger → 초기 난이도↑ → 보수적 복습 주기      |

**구현 핵심:**

- `ts-fsrs` npm 패키지 존재 (Python 공식 포팅). 직접 포팅 대신 이 패키지 검토
- 클라이언트 실행: 오프라인에서 "오늘 복습할 카드" 선별 가능하려면 FSRS 계산이 IndexedDB 데이터만으로 가능해야 함
- 서버와의 차이 보정: 동기화 시 서버 FSRS 결과가 최종 값

## 7. pdfplumber (PDF 파싱)

### 판정: ⚠️ 채택 (한글 테스트 필수)

**우리에게 맞는 이유:**

교재 835쪽 + 기출 PDF에서 텍스트와 표를 추출해야 합니다. pdfplumber는 Python 기반으로 표 추출에 특히 강합니다.

**잠재 위험:**

| 위험                                | 심각도   | 완화                                                             |
| ----------------------------------- | -------- | ---------------------------------------------------------------- |
| Workers에서 직접 실행 불가 (Python) | 중간     | 빌드 파이프라인(로컬/CI)에서만 실행. 런타임 아님                 |
| 한글 PDF 인코딩 문제                | **높음** | Phase 0에서 교재 PDF 실제 테스트 필수. 깨지면 PyMuPDF(fitz) 대안 |
| 스캔 PDF (이미지)                   | 높음     | pdfplumber는 텍스트 PDF 전용. 스캔본은 Claude Vision API(M04)    |
| 표 추출 정확도                      | 중간     | 교재의 복잡한 병합 셀 표에서 오류 가능. 수동 보정 프로세스 필요  |

**검토한 대안:**

- PyMuPDF(fitz): 더 빠름, 한글 지원 강함. 표 추출은 pdfplumber보다 약함
- pdf.js (JavaScript): Workers에서 실행 가능하나 표 추출 기능 없음
- Tabula: 표 특화, Java 의존
- Unstructured.io: AI 기반 파싱, 비용 발생

**결론:** pdfplumber로 시작하되, Phase 0 Step 0-4에서 실제 교재 PDF로 한글+표 추출 테스트. 실패 시 PyMuPDF로 즉시 교체. 두 라이브러리 모두 Python이라 교체 비용 낮음.

## 8. IndexedDB + Dexie.js (오프라인 스토리지)

### 판정: ✅ 채택 확정

**유일한 선택인 이유:**

브라우저에서 구조화된 데이터를 대량 저장할 수 있는 API는 IndexedDB뿐입니다. localStorage는 5MB 한도+동기식이라 학습 앱에 부적합.

**Dexie.js를 래퍼로 선택한 이유:**

- IndexedDB 네이티브 API는 콜백 기반으로 극도로 불편
- Dexie.js는 Promise 기반, 타입 안전, 인덱스 쿼리, 벌크 연산 지원
- ~26KB gzipped. 경량
- 오프라인 동기화 패턴(Dexie Cloud)도 제공하나, 우리는 자체 Background Sync 사용

**잠재 위험:**

| 위험                                  | 심각도   | 완화                                                           |
| ------------------------------------- | -------- | -------------------------------------------------------------- |
| iOS Safari IndexedDB 제한             | 중간     | iOS는 origin당 ~1GB. 우리 데이터 ~4MB면 충분                   |
| iOS PWA에서 7일 미사용 시 데이터 삭제 | **높음** | `navigator.storage.persist()` 요청 + 사용자에게 정기 접속 안내 |
| 동기화 충돌                           | 중간     | 타임스탬프 기반 LWW(Last Write Wins). 학습 데이터는 최신 우선  |

**대안 없음:** 브라우저 오프라인 스토리지에서 IndexedDB가 유일한 실용적 선택.

---

## 9. Zustand (상태 관리)

### 판정: ✅ 채택 확정

**Redux Toolkit 대신 Zustand인 이유:**

- 번들 ~1.2KB (RTK ~12KB). PWA 번들 최소화 목표와 일치
- 보일러플레이트 거의 없음. store 정의가 함수 하나
- React 외부에서 접근 가능 (`getState()`). Service Worker, Dexie.js 콜백에서 상태 읽기 필수
- devtools 미들웨어로 Redux DevTools 호환

**잠재 위험:**

| 위험                             | 심각도 | 완화                                                     |
| -------------------------------- | ------ | -------------------------------------------------------- |
| 복잡한 상태 로직에서 구조화 부족 | 낮음   | 우리 상태: session/progress/ui 3개 스토어. 단순함        |
| 서버 상태 관리 없음              | 낮음   | 서버 데이터 캐싱은 IndexedDB가 담당. Zustand은 UI 상태만 |

## 10. Hono (백엔드 프레임워크)

### 판정: ✅ 채택 확정

**Express/Fastify 대신 Hono인 이유:**

- Workers 런타임에 네이티브 설계. Web Standards API 기반
- 번들 ~14KB. Express는 Workers에서 아예 동작하지 않음
- 타입 안전 라우팅, Zod 밸리데이션 통합, OpenAPI 자동 생성
- D1/KV/R2 바인딩에 대한 타입 지원

**Hono 유일한 약점:** 생태계가 Express보다 작음. 하지만 우리는 미들웨어 2개(auth, error-handler)면 충분.

## 12. PWA + Background Sync + Web Push

### 판정: ✅ 채택 확정 (iOS 제한 인지)

**네이티브 앱 기각 사유:**

- 1 코드베이스 vs iOS+Android 별도 개발
- 스토어 심사 없이 즉시 업데이트
- 학습 앱 수준에서 PWA 성능 충분

**iOS 제한사항 (반드시 인지):**

| 제한                               | 영향                              | 완화                                             |
| ---------------------------------- | --------------------------------- | ------------------------------------------------ |
| Web Push: iOS 16.4+만 지원         | 복습 알림 미도달 가능             | 앱 내 알림 대체 + "홈 화면 추가" 유도            |
| PWA 7일 미사용 시 데이터 삭제 가능 | IndexedDB 데이터 유실             | `navigator.storage.persist()` + 서버에 진도 백업 |
| Background Sync: Safari 미지원     | 오프라인 학습 후 자동 동기화 불가 | 앱 재접속 시 수동 동기화 폴백                    |
| standalone 모드 제한               | 일부 Web API 제한                 | 핵심 기능(학습/복습)에는 영향 없음               |

**→ iOS 사용자 비율이 높을 경우 Phase 3에서 네이티브(React Native) 전환 검토. 현재는 PWA로 시작.**
