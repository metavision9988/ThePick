## 3. Phase 별 원문 발췌 (주요 증거만)

### Phase A (Surgeon — 바텀업)

> 🔴 Critical (신뢰도: 88): `rawBody.length`가 바이트 수가 아닌 JavaScript 문자열 코드포인트 수 — 파일:라인 202
> 문제: `rawBody.length > PAYLOAD_MAX_BYTES`에서 `PAYLOAD_MAX_BYTES = 32 * 1024`(32,768)와 비교하지만, `c.req.text()`는 UTF-8 디코딩된 문자열을 반환한다. 일본어/한국어/이모지 같은 멀티바이트 문자를 포함한 payload는 실제 바이트 수보다 `length`가 작게 측정된다.
> 수정: `new TextEncoder().encode(rawBody).byteLength > PAYLOAD_MAX_BYTES`로 교체하거나, `c.req.arrayBuffer()`로 받아 바이트 크기 먼저 체크 후 디코딩.

Devil's Advocate — `verifySignature` 가 항상 full HMAC 계산을 수행하는 timing-safe 설계는 확인됨. `providedSignature = ''` 경로도 `hexToBytes('')` 가 `null` 반환 → dummy compare 진입 확인.

### Phase B (Architect — 탑다운)

> B-1-Critical-2 (🔴): Drizzle `webhookEvents` 선언에 UNIQUE / INDEX / TRIGGER 누락 — SQL 과 1:1 불일치
> Drizzle 의 철학은 선언적 스키마 = SQL 소스. `drizzle-kit generate` 재생성 시 **UNIQUE 제약과 인덱스가 사라진 신규 마이그레이션이 산출**된다. 팀이 Phase 2 에 Drizzle generate 기반 마이그레이션으로 전환하면: (1) 새 마이그레이션에 `DROP INDEX idx_webhook_events_status`, `DROP ... UNIQUE` 포함 (2) 스테이징 적용 시 인덱스/제약 삭제 (3) Replay 공격 통과 (UNIQUE 없음), 쿼리 성능 저하

> B-2-Major-1 (🟠): 서명 검증 실패 시 body 파싱 미수행 — plan.md 명시 요구사항 위배 (Silent Pivot)
> plan.md §5 step 2 와의 불일치. 해결 옵션 (a) plan대로 body 파싱 추가 (b) ADR 업데이트

### Phase C (Contract — 도메인)

> M-1. ADR-002 addendum 미작성 (신뢰도 100)
> plan scope 에 명시된 인도물이 누락되었다. plan 검증 계획 8번 체크리스트 항목이 미완료 상태다.
> M-2. `rate-limit.ts` console.error/warn 잔존 — M-logger 해소 불완전 (신뢰도 88)
> Step 1-2 scope 의 M-logger 해소 목적은 auth 모듈의 모든 console.\* → logger 교체인데, routes.ts 와 hibp.ts 는 교체했으나 동일 auth 모듈인 rate-limit.ts 는 교체되지 않았다.

### Phase D (Advocate — 공격자)

> 🔴 Critical D-1-1. webhook 에 rate-limit 전무 — invalid signature CPU 소진 공격
> 공격자가 `/api/webhooks/payment/mock` 에 임의 signature + 32KB payload 를 초당 수천 회 전송. `verifySignature` 는 **항상 full HMAC-SHA256 계산** 을 수행한 뒤 timing-safe compare. Workers Free tier / Paid 모두 계정 단위 invocation 비용 bomb.

> 🔴 Critical D-1-2. payload 크기 검증이 HMAC 계산 이후에 수행됨
> 10MB payload → `c.req.text()` 로 10MB 적재 → HMAC 에 10MB 공급 → SHA-256(10MB) × 초당 1000건 = CPU DoS.
