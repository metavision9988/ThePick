## 🔴 Critical 9건 (중복 일부)

### 보안 (Pass 1 + Pass 3)

**C-1 (Pass 3 C1): Timing Side-Channel Enumeration** — `/login` 에서 `row === null` 경로는 verify 미실행(20ms) vs 이메일 존재 + 패스워드 불일치는 verify 실행(80~150ms). 공격자가 수험생 명부 식별 가능. **10만 이메일 × 50 RPS로 95%+ 식별률**.

- 완화: 더미 PBKDF2 verify (고정 상수 hash/salt) 항상 실행 또는 fixed-duration 지연

**C-2 (Pass 1 C2): suspended/deleted 분기 enumeration** — `row.status === 'suspended'` → 403 `ACCOUNT_SUSPENDED`, `'deleted'` → 403 `ACCOUNT_DELETED`. 이메일 존재 확인 경로.

- 완화: 둘 다 401 `genericFailure`로 통일 또는 정책 결정 명시

**C-3 (Pass 3 C2): Cache-Control Fail-Open** — FK PRAGMA 미들웨어가 early-return(`next()` 미호출) 시 cachePolicyMiddleware 실행 안 됨 → `/api/auth/*` 500 응답에 `Cache-Control` 누락.

- 완화: cachePolicyMiddleware를 **첫 번째**로 등록 + FK PRAGMA를 두 번째로

**C-4 (Pass 3 C3): 404/미매칭 경로 캐시 floor 없음** — `applyCachePolicy`가 매칭 실패 시 early return → 헤더 미설정.

- 완화: 말미에 기본 `Cache-Control: no-store` fallback 추가

**C-5 (Pass 3 C4): Rate Limiting 전무** — /register, /login 둘 다 rate limit 없음. 공격자 brute force + PBKDF2 310k iter 서버 부담으로 DoS 가능.

- 완화: Cloudflare Rate Limiting Rules (ADR-006 준수) — per-IP 10/min register, 20/min login + per-email 5-fail/10min

### 기획 대조 (Pass 4)

**C-6 (Pass 4 C1): ADR-005 PBKDF2 iterations 600k → 310k Silent Pivot** — ADR-005 본문 "600,000 (OWASP 2024)" vs 실제 구현/trigger 310000. `current.plan.md`가 "310000 (OWASP 2023)"로 암묵적 하향. ADR 공식 업데이트 또는 구현 상향 필요.

- 완화: ADR-005 Addendum 작성 "310k로 하향 + 근거(Workers CPU 50ms 마진)" **또는** 600k 상향

**C-7 (Pass 4 C2): v3.0 §7.1 `name` 컬럼 Silent Drop** — v3.0 `name TEXT` 컬럼 있음 / 실제 0006에 없음. ADR-007 §즉시반영 4가 구독 4컬럼만 강제하고 name은 침묵.

- 완화: 컬럼 추가(0007) 또는 ADR Year 2 이월 명시

### 아키텍처/코드 (Pass 1 + Pass 2)

**C-8 (Pass 1 C1): `timingSafeEqual` dummy === -1 구현 버그** — `dummy`는 0~255 범위, `-1` 반환 불가. 현재 호출 경로(verifyPassword)는 길이 가드로 진입 안 해 무해하나, public API라 미래 호출 위험 + 주석과 구현 불일치.

- 완화: `void dummy; return false;` 로 단순화 + 주석 교정

**C-9 (Pass 2 C1): users UPDATE 자동갱신 트리거 부재** — SQLite DEFAULT는 INSERT에만 적용. Step 1-3+ 에서 subscription/status UPDATE 구현자가 `updated_at` 누락 시 감사 추적 실패.

- 완화: `AFTER UPDATE ON users WHEN NEW.updated_at = OLD.updated_at BEGIN UPDATE users SET updated_at = datetime('now') WHERE id = NEW.id; END;` 트리거 추가 (재귀 주의 WHEN 절)

## Devil's Advocate (4개 에이전트 발언)

**Pass 1 반론**: PBKDF2 310k iter이 Workers free tier CPU 50ms 한도 초과 가능성 — Node.js 테스트로는 검증 불가

**Pass 2 반론**: FK PRAGMA 미들웨어 + withRetry 결합이 D1 부하 4배 증폭 → free tier 장애 루프 가능

**Pass 3 반론 A (즉시 공격)**: 수험생 명부 유출 — 10만 email 50 RPS login → timing + rate limit 없음으로 95%+ 식별률. 경쟁 학원/스팸 업자 공격 대상

**Pass 3 반론 B**: 기업 프록시 HIBP 차단 → 약한 비밀번호 가입 허용 → Phase 2 배치까지 방어선 없음

**Pass 4 반론**: ADR-005 자체가 "CPU 80% 연속 7일 → iterations 하향" 잘못된 기본값. 올바른 방어는 Argon2 전환 or rate limit 강화
