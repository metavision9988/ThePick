## 3. 수정 내역 (Critical 4 + scope Major 4)

### C-1 (D-6-2) — BAN 우회 방지

- `routes.ts /refresh` 에 `SELECT status FROM users WHERE id = ?` 재검증 추가 (routes.ts:449-465)
- `status !== 'active'` 시 `revokeAllUserSessions(userId)` + 401 `reason='user_not_active'`
- 신규 테스트: `BAN 우회 방지 (C-1 / D-6-2): user.status=suspended → 401 + 전체 세션 파기`

### C-2 (D-5-1 / B-2) — Refresh Rotation Grace Window

- `packages/shared/src/constants/auth.ts` — `REFRESH_ROTATION_GRACE_SECONDS = 60` 신규 상수
- `session.ts lookupRefreshSession` — revoked 세션의 `revoked_at` 이 60초 이내면 `reason: 'rotated_recently'` 반환 (reuse detection 회피)
- `routes.ts /refresh` — `rotated_recently` 는 일반 401, `revoked` 는 전체 세션 파기
- 신규 테스트:
  - session.test: `revokeSession + grace 이내 lookup → rotated_recently`
  - session.test: `revokeSession + grace 초과 lookup → revoked`
  - routes.test: `rotated_recently: grace 이내 재사용 → 401 + 전체 세션 유지`
  - routes.test 업데이트: reuse detection 테스트는 `revoked_at` 강제 과거화로 grace 초과 시나리오 구성

### C-3 (A-2) — NaN expires_at fail-safe

- `session.ts lookupRefreshSession:312` — `if (!Number.isFinite(expiresAt) || expiresAt <= now)` 로 전환. 스키마 손상/관리자 조작 시도 방어.

### C-4 (B-5 / B-1) — Drizzle drift 해소

- `migrations/0009_sessions.sql` — `refresh_token_hash TEXT NOT NULL UNIQUE` → `refresh_token_hash TEXT NOT NULL` + `CREATE UNIQUE INDEX IF NOT EXISTS sessions_refresh_token_hash_unique ON sessions(refresh_token_hash)` 로 분리
- `schema.ts:434` — partial index 에 `.where(sql\`revoked_at IS NULL\`)` 추가
- 롤백 스크립트에 DROP INDEX 추가

### M-1 (A-4) — signAccessToken fail-fast

- `session.ts:68-71` — 빈 userId/sessionId 시 `SIGN_EMPTY_USER_ID` / `SIGN_EMPTY_SESSION_ID` throw

### M-2 (A-1) — routes.ts:350 dead code 제거

- `const sessionId = lookup.ok ? lookup.sessionId : lookup.sessionId` → `const sessionId = lookup.sessionId` (union 공통 필드 접근)

### M-3 (D-7-1) — IP_PEPPER silent degradation 방지

- `routes.ts /login`, `/refresh` — pepper 미설정 시 `logger.warn('IP_PEPPER not configured — session.ip_hash will be null')`

### M-4 (B-2) — revokeAllUserSessions 재시도

- `routes.ts` — reuse detection + BAN 우회 경로의 `revokeAllUserSessions` 를 `withRetry()` 로 감쌈 (최대 3회 재시도)

### Devil's Advocate (최종 반론)

1. **"grace window 60초가 공격 창을 열지 않나?"** — 공격자가 정상 사용자의 rotation 직후 60초 이내 같은 refresh 를 확보하려면 MITM + HTTPS 복호화 필요. 이미 HttpOnly + Secure + SameSite=Strict 로 탈취 자체가 극히 어려움. 60초 내 탈취 성공 시에도 revoked 상태라 rotation 불가 (401). OAuth 2.0 Security BCP §4.12.1 업계 표준.

2. **"BAN 우회 방지에서 /refresh 에만 status 체크 — require-auth 미들웨어는?"** — Access JWT TTL 15min 이므로 BAN 후 최대 16분 (leeway 60s 포함) 창이 존재. TD-025 로 이월 등록하여 Phase 2 에 민감 write-path 라우트에 선택적 D1 session existence 체크 추가 권고.

3. **"grace 초과 reuse detection 시 revokeAllUserSessions withRetry 중 여전히 실패하면?"** — 3회 재시도 후에도 실패 시 logger.error 만. 차라리 500 반환하는 것이 안전하다는 주장 있으나, 공격자에게 401 이 아닌 500 을 주면 탈취 상태 힌트 제공. **401 유지 + 운영 알람**이 최적. Phase 2 에 Cloudflare Alerting 연동 (별도 ADR).
